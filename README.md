# Portable Ultrasonic DAQ System Build and Validation

This repository documents the design, integration, calibration, debugging, and validation of a portable 24-bit ultrasonic data acquisition system for broadband acoustic measurements. The system was built around an Avisoft ultrasonic sensor, the Analog Devices EVAL-AD4630-24FMCZ acquisition board, a Digilent ZedBoard, Ethernet-based data transfer, Python-based capture control, and MATLAB-based analysis.

The primary purpose of this repository is to document the full engineering workflow behind the system: hardware selection, signal-chain design, acquisition architecture, calibration, frequency-response diagnosis, digital compensation, and ongoing anti-alias filter integration. The focus is not only on the final working system, but also on the reasoning behind each major design decision and the validation steps used to confirm performance.

---

## Project Objective

The objective of this build was to create a portable and practical ultrasonic DAQ platform capable of:

- high-resolution broadband acquisition
- battery-powered operation
- reliable field-oriented data capture
- calibrated voltage reconstruction
- compensation of known analog front-end limitations
- direct handoff into downstream signal-processing workflows

The target measurement regime is broadband ultrasonic response generated by impulsive excitation, such as hammer-strike testing, where waveform fidelity, bandwidth, and repeatability matter.

### Core hardware
- **Sensor:** Avisoft CM16/CMPA-5V, 2 to 200 kHz, balanced differential output
- **ADC board:** EVAL-AD4630-24FMCZ (Rev E), 24-bit simultaneous sampling
- **Processing platform:** Digilent ZedBoard with Zynq-7020 (pre-built ADI HDL reference design)
- **Communication link:** Gigabit Ethernet
- **Power concept:** portable battery-powered operation with isolated analog and digital domains

### Core software
- **Python:** hardware communication, remote execution, file transfer, raw-to-processed handoff
- **MATLAB:** calibration, time-domain analysis, frequency-domain analysis, and validation

## Design Philosophy

This build was driven by a few core engineering requirements:

### 1. Keep the acquisition path robust
The data capture path was designed around local acquisition on the ZedBoard instead of live network streaming. This avoids network-induced sample continuity problems and keeps the acquisition side simple and reliable.

### 2. Preserve measurement quality
Ethernet was preferred over USB because it provides galvanic isolation through standard PHY magnetics, which helps reduce unwanted coupling between the host computer and the precision measurement hardware.

### 3. Separate acquisition from analysis
Python was used for hardware interfacing and capture control. MATLAB was kept as the main analysis environment because it is better suited for the downstream signal-processing workflow already used in this project.

### 4. Validate problems before fixing them
Observed bandwidth loss was not treated as a software issue by default. The system was tested, modeled, and fitted until the analog front-end bottleneck was confirmed quantitatively before compensation was introduced.

## Key Engineering Results

### Calibrated voltage conversion
A 9-point DC calibration established the raw-count to voltage mapping:

- Gain = `0.0000006476 V/count`
- Offset = `0.148989 V`
- Calibration R² ≈ `0.99998`

Measured performance:
- ENOB ≈ `16.9 bits`
- SNR ≈ `103.2 dB`
- Broadband noise floor ≈ `20 µV RMS`

### Analog front-end rolloff identified
During validation, signal amplitude was found to roll off above roughly 30 kHz. Analysis of the eval-board analog front end and subsequent system identification showed that the dominant limitation came from the ADA4945-1 input RC network on the board.

Results:
- Predicted corner frequency from schematic values: `55.6 kHz`
- Measured best-fit single-pole corner frequency: `53.5 kHz`

The close agreement between predicted and measured values confirmed that the rolloff was a real hardware limitation in the analog front end.

### Frequency-domain compensation implemented
A zero-phase frequency-domain compensation method was implemented to correct the measured analog front-end response without introducing startup transients or phase distortion.

Compensation model:

$$H_{comp}(f) = \frac{1 + j \cdot f / f_c}{1 + j \cdot f / f_l}$$

where $f_c = 53{,}500$ Hz (measured pole) and $f_l = 500{,}000$ Hz (boost limit).

This approach was selected because:

- it avoids the doubled-magnitude effect that occurs with `filtfilt`
- it avoids startup transient corruption from causal IIR filtering
- it gives direct control over the compensation shape in the frequency domain
- it preserves zero phase distortion

### MATLAB compensation bug found and fixed
During square-wave validation, a MATLAB implementation issue was identified in the handling of FFT bins above N/2. Those bins were being treated as positive frequencies instead of mirrored negative frequencies, which caused incorrect compensation and large spikes in the reconstructed waveform. Rebuilding the transfer function with proper conjugate symmetry fixed the issue and aligned MATLAB output with the Python implementation.

## Workflow Summary

1. Boot and configure the ZedBoard + AD4630 acquisition hardware
2. Capture raw binary data locally on the ZedBoard
3. Transfer the capture to the Windows host over Ethernet
4. Convert raw counts to calibrated voltage
5. Apply measured frequency-response compensation
6. Validate the result in time and frequency domains
7. Use the processed output in downstream signal-analysis workflows
