Standard Operating Procedures — Ultrasonic Data Acquisition

1
Power-On Sequence
Step 1 Verify Physical Connections

Before powering anything, confirm all of the following:

  • SD card seated in ZedBoard slot
  • Ethernet cable: PC ↔ ZedBoard RJ-45 Gigabit Ethernet jack (confirm the connector designator against your ZedBoard revision silkscreen)
  • ZedBoard power cable: DC5525 into J20
  • Boot jumpers confirmed: JP7=GND, JP8=GND, JP9=3V3, JP10=3V3, JP11=GND
  • VADJ set to 2.5V (do not change)
Step 2 Power On ZedBoard
  • Flip SW8 ON
  • Wait 30 seconds for full boot
Step 3 Open Serial Terminal (PC)
  • PuTTY → Serial → your board's COM port → 115200 baud
  • Press Enter if no prompt appears
  • Log in with the credentials configured on your ZedBoard image. The stock Kuiper image ships with vendor defaults; change them before deployment or before connecting to an untrusted network.
Step 4 Verify Network

Run on ZedBoard serial terminal:

bash 
ip addr show eth0
✓ Expected: shows 192.168.1.100

If not shown, assign the IP manually:

bash 
sudo ip addr add 192.168.1.100/24 dev eth0
sudo ip link set eth0 up
ℹ This is a temporary recovery address. It is removed at reboot. After restoring access, configure the persistent address using the network method supported by the installed Kuiper image.
ℹ Administrative commands in this guide require root privileges. When logged in as a non-root account such as analog, prefix commands such as ip and systemctl start/restart with sudo.
Step 5 Verify iiod is Running
bash 
systemctl status iiod --no-pager
✓ Expected: active (running)

If not running, start it:

bash 
sudo systemctl start iiod
Step 6 Verify ADC is Visible
bash 
iio_info -u local: | grep ad4630
✓ Expected: returns the device name
⚠ If nothing returns, first confirm that Kuiper Linux completed booting, the SD card is seated, the evaluation board is fully seated on the FMC connector, VADJ is 2.5 V, and the required FPGA configuration and device tree loaded. Power-cycle only after correcting the hardware or boot problem.
Step 7 Health Check — Confirm DMA is Live
bash 
rm -f /tmp/healthcheck.bin
iio_readdev -u local: -b 4096 -s 4096 ad4630-24 > /tmp/healthcheck.bin
stat -c "%s bytes" /tmp/healthcheck.bin
od -An -tx4 -N64 /tmp/healthcheck.bin
rm -f /tmp/healthcheck.bin
✓ Expected: /tmp/healthcheck.bin is 32768 bytes (4096 frames × 2 channels × 4 bytes)
⚠ Expected size: 32768 bytes. The od output should contain changing hexadecimal words rather than only zeros. If the command hangs or the result is still invalid, recheck the ADC visibility and hardware setup, then use the staged recovery sequence below.
Step 8 Verify PC Network
  • PC IP should be 192.168.1.10 on the wired Ethernet adapter connected to the ZedBoard

Ping test from PC command prompt:

cmd / powershell 
ping 192.168.1.100
✓ Expected: all packets return ✗ If not: check Ethernet adapter settings
Step 9 Verify Host Network-IIO Connectivity

Run from a host terminal:

cmd / powershell / bash 
iio_info -u ip:192.168.1.100
✓ Expected: the output lists the ad4630-24 device and its channels
ℹ This uses the network-IIO interface (iiod, port 30431) and is a diagnostic check only. Normal captures run iio_readdev locally on the ZedBoard and transfer the completed file through SSH and SFTP on port 22.
2
Shutdown Sequence
ℹ Normal shutdown uses python scripts/shutdown_zedboard.py from the repository root. The script issues the remote poweroff and uses consecutive failed pings as its shutdown criterion. Confirm the final board state using the serial console or board indicators when practical. The manual serial sequence below is the fallback for when the host or network path is unavailable.
🛑 Never just flip SW8 off — always perform a clean shutdown first.
Step 1 Initiate Shutdown from ZedBoard Serial Terminal
bash 
sudo shutdown -h now
Step 2 Wait for Halt Confirmation

Watch the serial terminal for:

terminal output 
Reached target Shutdown
ℹ The terminal may also go silent / unresponsive. This takes approximately 15–20 seconds.
Step 3 Cut Power

Wait an additional 5 seconds after halt message, then flip SW8 OFF.

Step 4 Power Off Instruments

Power off the signal generator and any other bench instruments.

Step 5 Full De-energize (Optional)

Unplug both battery packs (the ZedBoard 12 V pack and the isolated 5 V sensor supply).

3
Reset / Recovery Sequence

Use this procedure when any of the following occur:

  • DMA buffer locks
  • All captures return zeros
  • iiod becomes unresponsive
  • Python hangs on adc.rx()
⚡ Level 1a — Try First
Network-IIO Recovery Only Use only when iiod diagnostics fail
ℹ Use this only when iio_info -u ip:192.168.1.100 or another network-IIO diagnostic fails while local IIO remains available. Restarting iiod does not repair normal local iio_readdev captures.
bash 
sudo systemctl restart iiod

Then repeat Step 9 to verify network-IIO connectivity. If the local finite capture in Step 7 fails, continue to the Linux reboot or full power-cycle recovery levels instead.

↻ Level 1b — If 1a Fails
Full Linux Reboot Preferred over power cycle
bash 
sudo reboot
💡 Wait 30 seconds for full reboot, then re-run Power-On Sequence from Step 3 onward. A reboot reloads the Linux drivers and usually reconfigures the FPGA during boot, but it does not guarantee a full programmable-logic reset. If the ADC is still not visible or the health check fails, use the Level 2 power cycle, which fully de-powers the FPGA.
⚠ Level 2 — If Level 1 Fails
Full Power Cycle
  1. From serial terminal, initiate shutdown:
bash 
sudo shutdown -h now
  1. Wait for halt confirmation in terminal
  2. Flip SW8 OFF
  3. Wait 15 seconds minimum — capacitors discharge, FPGA fully de-powers
  4. Flip SW8 ON
  5. Wait 30 seconds for full boot
  6. Re-run the full Power-On Sequence from Step 3 onward
  7. Run health check (Step 7) before any Python captures
🔴 Level 3 — If Power Cycle Fails
SD Card / FPGA Check
  • Re-seat the SD card (power off first)
  • Check that FPGA bitstream is loading — boot LEDs should sequence on ZedBoard during boot
  • Re-run full Power-On Sequence

Key Constants — Quick Reference
Parameter Value
ZedBoard IP192.168.1.100
PC IP192.168.1.10
COM Port115200 baud (set your board's COM port)
Ethernet PortZedBoard RJ-45 Gigabit Ethernet (verify designator on your board)
Power InputZedBoard J20, DC5525, 12V
VADJ2.5V — never change
Sample Rate500 kSPS per channel (dual-channel, set directly)
Sample Frametwo signed 32-bit words/frame; 24-bit result in upper bits (parse: reshape(-1, 2), then >> 8)
Working Directoryhost capture directory (set in script config)
Boot JumpersJP7=GND, JP8=GND, JP9=3V3, JP10=3V3, JP11=GND
ℹ The device is configured to 500 kSPS per channel directly. Each capture is parsed as two-channel frames (reshape(-1, 2)) followed by the signed 24-bit shift (>> 8). There is no 1 MSPS hardware rate or [::2] decimation step.
